# 전수검사 모드 (Total Inspection)

당신은 지금 외주 결과물의 **전수검사**를 수행합니다.
이것은 일반 검수보다 훨씬 강력하며, **빠짐없이 모두 검사**하는 것이 핵심입니다.

검사 범위: $ARGUMENTS (지정 없으면 전체 프로젝트)

---

## 작업 원칙 (반드시 준수)

1. **샘플링 금지** — 모든 파일, 모든 함수, 모든 엔드포인트, 모든 페이지를 검사합니다.
2. **누락 보고 의무** — 검사하지 못한 영역이 있으면 명시적으로 보고합니다.
3. **가차 없이** — 의심스러우면 보고합니다. False positive는 사용자가 판단합니다.
4. **외주 책임 한계 보호** — 발견된 모든 이슈가 클라이언트와의 분쟁 가능성을 의미합니다.
5. **모델 비용 < 누락 비용** — 시간이 걸리더라도 빠짐없이.

---

## 실행 절차

### Step 0 — 사전 점검 (직접 수행)
1. 프로젝트 루트의 `CLAUDE.md` (기획문서) 존재 확인
   - 없으면 사용자에게 보고하고 중단: "기획문서 없이 전수검사는 의미가 약합니다."
2. 프로젝트 구조 파악:
   - `git ls-files` 또는 `find . -type f` 로 전체 파일 수 카운트
   - 주요 디렉토리 (src, app, pages, api, lib, components 등) 식별
   - 사용 기술 스택 확인 (package.json, requirements.txt 등)
3. 검사 계획 수립 후 사용자에게 짧게 보고:
   - "총 N개 파일, M개 모듈을 8개 + 코드품질 인라인 에이전트로 검사합니다. 예상 소요 시간: 약 X분."

### Step 1 — 8개 전문 에이전트 병렬 스폰 (★ v2.0 핵심)

**한 메시지 안에서 모든 에이전트를 동시에 호출합니다.** (병렬 실행)

다음 8개 에이전트를 동시 스폰:

1. **plan-compliance-checker** (Opus) — 기획 준수 검사 (가장 중요)
   - 프롬프트: "프로젝트 루트의 CLAUDE.md 기획문서를 기준으로 In-Scope 누락과 Out-of-Scope 침범을 전수 점검. 검사 범위: $ARGUMENTS"

2. **security-checker** (Opus) — 보안 전수검사 (책임 직결)
   - 프롬프트: "OWASP Top 10 + 시크릿 노출 + 인증·인가를 빠짐없이 점검. 모든 API 엔드포인트, 모든 입력 처리 지점, 모든 .env/config 파일 점검. 검사 범위: $ARGUMENTS"

3. **accessibility-checker** (Sonnet) — 접근성 전수검사
   - 프롬프트: "WCAG 2.1 AA 기준으로 모든 페이지·컴포넌트 점검. 키보드, 스크린리더, 색대비 모두. 검사 범위: $ARGUMENTS"

4. **performance-checker** (Sonnet) — 성능 전수검사
   - 프롬프트: "번들 크기, 렌더링 최적화, N+1 쿼리, Core Web Vitals 모두 점검. 가능하면 빌드 후 측정. 검사 범위: $ARGUMENTS"

5. **ux-reviewer** (Sonnet) — UX 전수검사
   - 프롬프트: "모든 주요 사용자 흐름을 시나리오로 시뮬레이션. 에러 메시지, 로딩 상태, 카피, UI 일관성 점검. 검사 범위: $ARGUMENTS"

6. **기능 테스트** — general-purpose Agent ★ (v2.0 신설)
   - 프롬프트: "프로젝트 루트 CLAUDE.md의 In-Scope 각 항목에 대해 실제 동작 가능 여부를 검증. 가능하면 dev 서버 띄우고 curl/headless 브라우저로 골든 패스 + 경계 조건 + 에러 복구 시뮬레이션. 검사 범위: $ARGUMENTS"

7. **사용자 시나리오** — general-purpose Agent ★ (v2.0 신설)
   - 프롬프트: "기획문서의 페르소나 기반으로 일상 사용 시나리오 5개를 자동 생성해 끝까지 시뮬레이션. 각 시나리오에서 마찰점·혼란 지점·버그 가능성을 보고. 검사 범위: $ARGUMENTS"

8. **회귀** — general-purpose Agent ★ (v2.0 신설)
   - 프롬프트: "최근 commit 30개의 diff를 훑고 기존 기능에 영향이 갈 만한 변경을 찾는다. 특히 공통 모듈·유틸·시그니처 변경에 주목. 검사 범위: $ARGUMENTS"

9. **+ code-quality 인라인 검사** — Agent(Explore 또는 general-purpose)로 별도 스폰
   - 프롬프트: "다음을 빠짐없이 점검하세요:
     - 모든 파일에 대한 dead code, 중복 코드, 매직 넘버
     - 모든 함수의 단위 테스트 존재 여부 (테스트 파일 매칭)
     - 모든 의존성에 대한 npm/pnpm/pip audit 실행
     - 모든 TODO/FIXME 주석 수집
     - 모든 console.log / print 디버그 코드 잔존 여부
     - 모든 환경변수 사용처가 .env.example에 정의되어 있는지
     - 모든 라우트/엔드포인트 목록 추출
     - 모든 데이터베이스 마이그레이션 파일과 실제 스키마 일치 여부
     검사 범위: $ARGUMENTS"

### Step 2 — 결과 통합 분석

모든 에이전트 결과를 받은 뒤:

1. **중복 이슈 통합** — 여러 에이전트가 같은 문제를 보고했으면 머지
2. **우선순위 재정렬** — 외주 책임 관점에서 재평가
   - 🔴 Critical: 인수 거절 / 보안 사고 / 법적 책임 가능성
   - 🟠 High: 클라이언트 클레임 가능성 / 출시 후 즉시 수정 필요
   - 🟡 Medium: 다음 스프린트
   - 🟢 Low: 백로그
3. **상호 영향 분석** — 한 이슈를 고치면 다른 이슈에 영향 있는지

### Step 3 — 최종 보고서 생성

다음 형식으로 사용자에게 출력:

```markdown
# 전수검사 보고서
검사 일시: YYYY-MM-DD HH:MM
검사 범위: $ARGUMENTS
검사 파일 수: N / 전체 M (커버리지 X%)

## 📊 한 줄 요약
[배포 가능 / 배포 불가 / 조건부 배포 가능] — 핵심 이유 한 줄

## 🚨 Critical (즉시 수정, 배포 불가)
N건
1. [에이전트명] [파일:라인] 이슈 제목
   - 발견 내용:
   - 영향: (외주 책임/보안/사용자 영향)
   - 수정 방안: (구체적 코드 또는 단계)

## 🟠 High (배포 전 수정)
N건
...

## 🟡 Medium (출시 후 1주 내)
N건
...

## 🟢 Low / 양호
- 잘 적용된 부분 요약

## 📋 클라이언트 인수 체크리스트
[ ] 항목 1
[ ] 항목 2
...

## ⚠️ 검사 한계 (빠짐없이 명시)
- 검사하지 못한 영역: ...
- 자동 검증 불가능한 항목: ...
- 사용자가 직접 확인 필요한 부분: ...

## 다음 액션
1. ...
2. ...
```

### Step 4 — 보고서 파일 저장

보고서를 다음 경로에 저장:
`./inspection-report-YYYY-MM-DD-HHMM.md`

이렇게 하면 클라이언트 보고에 그대로 첨부 가능.

---

## 절대 하지 말 것

- "샘플 몇 개만 봤는데 괜찮아 보입니다" 식 결론 금지
- 에이전트 하나라도 빼먹기 금지
- "시간 관계상 생략" 금지 — 시간이 걸려도 끝까지
- False positive 두려워서 보고 누락 금지
