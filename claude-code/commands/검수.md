# 표준 검수 (4단계)

검수 범위: $ARGUMENTS (지정 없으면 마지막 커밋 이후 변경분)

## 절차

### Step 1 — 변경분 파악
- `git diff` 또는 `git log --oneline -10`으로 변경 범위 식별
- 사용자에게 검수 대상 확인

### Step 2 — 핵심 4개 에이전트 병렬 스폰

다음을 한 메시지 안에서 동시 호출:

1. **plan-compliance-checker** — 변경분이 기획문서를 위반하지 않는지
2. **security-checker** — 변경분의 보안 점검 (변경된 엔드포인트/입력 처리 위주)
3. **performance-checker** — 변경분이 성능에 미치는 영향
4. **ux-reviewer** — 변경분의 사용자 영향

### Step 3 — 통합 보고

전수검사와 동일한 우선순위 체계 (Critical/High/Medium/Low).
다만 검사 범위가 좁아 보고서도 짧게.

## 전수검사와의 차이 (v2.0)

| 구분 | /검수 | /전수검사 |
|---|---|---|
| 범위 | 변경분 | 전체 프로젝트 |
| 에이전트 수 | **4개** (plan-compliance·security·performance·ux-reviewer) | **8개 + 코드품질** (위 4 + accessibility + 기능 테스트 + 사용자 시나리오 + 회귀 + code-quality 인라인) |
| 접근성 검사 | 변경된 페이지만 (별도 에이전트 없음, ux-reviewer가 겸함) | accessibility-checker 전용 |
| 기능 테스트·사용자 시나리오·회귀 | 안 함 | 함 (v2.0 신설) |
| 코드 품질 인라인 | 안 함 | 함 (dead code, audit 등) |
| 보고서 저장 | 채팅만 | 파일로도 저장 (`./reports/inspection-report-*.md`) |
| 적합 시점 | 일상 작업, 모듈 commit 후 | §5 검수, 배포 직전, 마일스톤 종료 |

## 사용자가 더 강한 검수를 원하면

`/전수검사`를 안내합니다.
