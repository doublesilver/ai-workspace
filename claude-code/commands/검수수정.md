# 검수 + 자동수정 통합 모드

당신은 지금 외주 결과물의 **전수검사 → 자동수정 → 재검증**을 한 흐름으로 수행합니다.
`/전수검사`와 `/자동수정`을 묶은 풀 사이클이며, 개별 호출을 두 번 하는 대신 끊김 없이 실행합니다.

인자: $ARGUMENTS
- 우선순위 필터: `all`(기본) / `critical만` / `critical+high` / `critical+high+medium`
- 범위: 별도 지정 없으면 전체. 예: `src/api`, `결제 모듈`, `pages/checkout`

---

## 작업 원칙 (반드시 준수)

1. **샘플링 금지** — 모든 파일·함수·엔드포인트·페이지 전수 검사.
2. **누락 보고 의무** — 검사하지 못한 영역은 명시.
3. **외주 책임 한계 보호** — 발견된 이슈 = 분쟁 가능성.
4. **모델 비용 < 누락 비용** — 시간이 걸려도 빠짐없이.
5. **무한 루프 방지** — 검수 → 수정 → 재검증 사이클은 **최대 3회**.

---

## 실행 절차

### Step 0 — 사전 점검
1. 프로젝트 루트의 `CLAUDE.md` (기획문서) 존재 확인.
   - 없으면 중단: "기획문서 없이는 In-Scope 판정이 불가능합니다."
2. 인자 파싱:
   - 우선순위 키워드(`critical만`/`critical+high`/`critical+high+medium`/`all`)와 범위를 분리.
   - 키워드가 없으면 우선순위 = `all`.
3. 프로젝트 구조 파악(`git ls-files` 또는 `find . -type f`)하고 사용자에게 짧게 보고:
   - "총 N개 파일, M개 모듈. 8개 + 인라인 에이전트로 검수 후 발견 이슈를 자동 수정합니다. 최대 3사이클."

### Step 1 — 전수검사 (`/전수검사`와 동일)

**핵심: 8개 전문 에이전트 + code-quality 인라인 검사를 한 메시지 안에서 병렬 스폰합니다.**

스폰할 에이전트 목록과 프롬프트는 `/전수검사`의 Step 1과 동일하게 사용:

1. **plan-compliance-checker** (Opus) — In-Scope 누락 / Out-of-Scope 침범 전수 점검.
2. **security-checker** (Opus) — OWASP Top 10 + 시크릿 + 인증·인가 전수 점검.
3. **accessibility-checker** (Sonnet) — WCAG 2.1 AA 전수 점검.
4. **performance-checker** (Sonnet) — 번들·렌더링·N+1·Core Web Vitals 전수 점검.
5. **ux-reviewer** (Sonnet) — 모든 사용자 흐름 시뮬레이션.
6. **기능 테스트** (general-purpose) — In-Scope 각 항목 동작 검증, dev 서버 + curl/headless.
7. **사용자 시나리오** (general-purpose) — 페르소나 기반 5개 시나리오 시뮬레이션.
8. **회귀** (general-purpose) — 최근 commit 30개 diff 기반 회귀 위험 탐지.
9. **code-quality 인라인** (Explore 또는 general-purpose) — dead code/중복/매직넘버/audit/TODO/console.log/.env 정합성/라우트 목록/마이그레이션 정합성.

(각 에이전트에 전달할 검사 범위 = Step 0에서 파싱한 "범위" 인자. 없으면 전체.)

### Step 2 — 결과 통합 및 보고서 생성

모든 에이전트 결과 수신 후:
1. **중복 머지** — 여러 에이전트가 같은 문제를 보고했으면 통합.
2. **우선순위 재정렬**:
   - 🔴 Critical — 인수 거절 / 보안 사고 / 법적 책임
   - 🟠 High — 클라이언트 클레임 / 출시 후 즉시 수정
   - 🟡 Medium — 다음 스프린트
   - 🟢 Low — 백로그
3. **상호 영향 분석**.
4. 보고서를 `./inspection-report-YYYY-MM-DD-HHMM.md`로 저장 (`/전수검사`의 Step 3 형식).
5. 사용자에게 한 줄 요약 출력:
   - 예: "Critical 2 / High 5 / Medium 8 / Low 3 발견. 자동 수정 진입합니다."

### Step 3 — 조기 종료 판정

- 우선순위 인자에 해당하는 이슈가 **0건이면** Step 4를 건너뛰고 Step 7로 직행 (성공 종료).
  - 예: 인자가 `critical만`인데 Critical 0건 → 자동수정 불필요.
- 그 외에는 Step 4 진입.

### Step 4 — 자동 수정 (`/자동수정`과 동일)

`/자동수정`의 Step 2~3 절차를 그대로 따릅니다:

1. **TaskCreate** — 인자 우선순위에 해당하는 모든 이슈를 task로 등록. prefix는 `[Critical]`/`[High]`/`[Medium]`/`[Low]`. 처리 순서는 항상 **Critical → High → Medium → Low**. 의존성 있으면 `addBlockedBy`.
2. **자동 수정 실행** — 각 task별:
   - TaskUpdate status=in_progress
   - 보고서의 권장 수정안에 따라 코드 수정
   - 단위 테스트 또는 빌드로 즉시 검증
   - 통과 시 자동 commit (메시지: `fix: [이슈 제목]`)
   - TaskUpdate status=completed
3. 막히는 task는 in_progress 유지하고 다른 task 계속 진행. 종료 시 막힌 task 모음 보고.

### Step 5 — 재검증 (★ 핵심)

수정 사이클이 끝나면 **Step 1 전수검사를 다시 실행**합니다.

- 새 보고서 생성 → 이전 보고서와 비교:
  - 수정된 이슈: ✅
  - 남은 이슈: ⚠️
  - 새로 발견된 이슈(회귀): 🆕

- 남은 이슈가 있고 사이클 카운트 < 3:
  - 우선순위 필터 범위 내에서 남은 이슈가 있으면 Step 4로 **루프 백** (사이클 카운트 +1).
- 사이클 카운트 3 도달 또는 남은 이슈 0:
  - Step 6 진입.

### Step 6 — 최종 commit 및 push

- 사이클별 commit이 누적돼 있으면 그대로 유지. 사이클 마지막에 변경이 남아있으면 일괄 commit.
- 메시지 예: `fix: 검수수정 사이클 N — Critical X / High Y / Medium Z / Low W 해결`
- 글로벌 §13.1 정책에 따라 자동 push (non-main 브랜치만).

### Step 7 — 종합 보고

다음 형식으로 사용자에게 출력:

```markdown
## 검수수정 결과 (사이클 N회)

### 최종 상태
- 🔴 Critical: 시작 A건 → 남은 B건
- 🟠 High: 시작 C건 → 남은 D건
- 🟡 Medium: 시작 E건 → 남은 F건
- 🟢 Low: 시작 G건 → 남은 H건

### 자동 수정 성공
- 사이클 1: N건
- 사이클 2: M건
- 사이클 3: K건 (해당 시)

### 자동 수정 실패 (사용자 결정 필요)
1. [이슈] — 사유: ...

### 회귀 (수정 중 새로 발생)
1. [이슈] — 위치: ...

### 다음 액션
- 남은 Critical/High 0건 → §6 UAT 또는 배포 가능
- 그 외 → 사용자 개입 필요 항목 요약
```

### Step 8 — UAT stale 알림 (★ v2.0)

코드 수정이 실제 발생했고 `./uat/*결과*.md` 파일이 존재하면 알림:

> "코드가 변경됐습니다. 이전 UAT 결과는 stale 상태입니다.
> 1. `/UAT`로 새 시나리오 생성 → 클라이언트 재검수, 또는
> 2. 영향 범위가 좁다면 기존 결과 파일에서 해당 항목만 ❌로 변경 후 재검수"

보고서에 "UAT 재실행 필요" 항목 추가.

## 절대 하지 말 것

- "샘플 몇 개만 봤는데 괜찮아 보입니다" — 금지.
- 에이전트 1개라도 빼먹기 — 금지.
- 3사이클 초과 자동 진행 — 금지 (사용자 개입 필요).
- 검수 보고서 저장 누락 — 금지 (재검증 비교 불가능해짐).
- Critical/High 남은 채로 "배포 가능" 결론 — 금지.

## 참고

- 검수만 하고 싶으면 `/전수검사`.
- 이미 검수 보고서가 있고 수정만 돌리고 싶으면 `/자동수정`.
- 두 단계를 끊김 없이 풀 사이클로 돌릴 때 이 커맨드(`/검수수정`)를 사용.
