# 8단계 — 인수·하자보수 시작

당신은 외주 프로젝트의 **공식 인계 문서**를 생성합니다.
**선행 조건**: 6단계 UAT 통과 + 7단계 Production 배포 완료.

## 절차

### Step 1 — 사전 점검
- `./uat/` 최신 파일에 미통과 항목 0건 확인
- Production 배포 commit/tag 확인 (`git log`로 마지막 production deploy 흔적)
- 위 둘 중 하나라도 미충족이면 정지 후 사용자 보고

### Step 2 — 입금 상태 확인
`./contracts/입금트래커.md` 읽기:
- 착수금·중도금 입금 확인 (잔금은 인수확인서 서명 후 청구)
- 미입금 단계가 있으면 사용자에게 알림

### Step 3 — 인수확인서 작성

`./acceptance/YYYY-MM-DD-인수확인서.md`:

```markdown
# {프로젝트명} 인수확인서

작성일: YYYY-MM-DD
시공자: {본인 정보 — 사업자명·연락처}
발주자: {클라이언트 정보}

## 1. 인수 범위
다음 산출물의 인수를 확인합니다:
- 소스코드 (저장소: {git repo URL or zip 인계 방식})
- **테스트 스위트 (`tests/` — §15.1 회귀 자산)**
- 기획문서 (`CLAUDE.md`)
- 디자인 산출물 (`docs/design/`)
- 인계 매뉴얼 (자동 생성된 학습 노트 모음)
- Production 배포 (URL: {production URL})
- 환경 변수 목록 + 외부 서비스 계정 인계

## 2. 인수 시점 정의
다음 모두 충족 시 인수 완료:
- [x] UAT 체크리스트 전 항목 통과 ({./uat/...} 참조)
- [x] Production 배포 완료 (YYYY-MM-DD)
- [x] 인계 매뉴얼 전달
- [ ] 발주자 서명 (이 문서 하단)

## 3. 하자보수
**기간**: YYYY-MM-DD ~ YYYY-MM-DD (총 N일, 통상 30~90일)

**무상 범위 (하자보수)**:
- 인수 시점에 명시된 In-Scope 기능의 버그
- 정상 사용 환경에서의 오작동
- 보안 패치 (Critical/High 발견 시)

**유상 범위 (별도 견적)**:
- Out-of-Scope 기능 추가
- 디자인 변경
- 데이터 마이그레이션·튜닝
- 사용자 수/매장 수 확장에 따른 인프라 변경
- 클라이언트의 인프라·외부 서비스 변경에 따른 재작업

## 3-1. SLA (서비스 수준 약정) — §15.3
하자보수 기간 동안 적용:

| 항목 | 약정값 | 비고 |
|---|---|---|
| 가동률(uptime) | {99.5% 기본 / 인프라 등급 따라 조정} | 무료·저가 인프라는 99.5%, Vercel Pro·전용서버는 상향 |
| 장애 1차 응답 | critical 영업일 4h / 일반 영업일 24h | |
| 모니터링 | UptimeRobot(가동률) + Sentry(에러) | Step 5에서 셋업 |

- SLA는 **하자보수 기간 내에서만 무상**. 이후는 별도 운영 계약(월 정액)으로 전환.
- 측정 가능한 약정이 어려운 소규모 건은 **"best-effort·무보장"임을 여기에 명문화** (그것도 SLA의 일종).
- SLA 위반(다운타임 초과 등) 시 대응 절차와 책임 한계도 본 항에 함께 기재.

## 4. 발주자 책임
- 환경 변수·API 키 자체 관리 (인계 후)
- 외부 서비스 결제 (Vercel/Railway/Sentry 등)
- 사용자 데이터 관리 (개인정보·결제 정보)

## 5. 잔금 청구
- 금액: ₩... (계약서 기준 30%)
- 입금 마감: 본 문서 서명일 + 7일
- 입금 계좌: {...}

## 6. 서명

발주자 서명: ___________  날짜: ___________
시공자 서명: ___________  날짜: ___________
```

### Step 4 — 인계 매뉴얼 통합
개발 중 누적된 학습 노트(4단계 누적분)를 한 파일로 정리:

`./acceptance/인계매뉴얼.md`:
- 프로젝트 구조 한눈에
- 주요 환경 변수 + 의미
- 자주 쓰는 명령어 (개발/배포/롤백)
- 외부 서비스 계정 인계 체크리스트
- 운영 시 자주 보는 대시보드 링크

### Step 5 — 모니터링 셋업 안내
하자보수 기간 동안 자동 감지가 가능하도록:
- Sentry 또는 동등 도구 권장
- Vercel Analytics / Railway logs 알림 셋업
- Uptime 모니터링 (UptimeRobot 등 무료 옵션)

상세 셋업은 `/하자보수`로 진행 안내.

### Step 6 — 클라이언트 전달
한국어 존댓말 카카오톡·이메일 메시지:

> 안녕하세요, {프로젝트명} 개발이 마무리되어 공식 인계를 진행합니다.
> 첨부드린 **인수확인서**를 확인하신 뒤 서명 부탁드립니다.
> 서명일부터 {N}일간 무상 하자보수가 적용됩니다.
> 잔금은 서명일 + 7일 내로 입금 부탁드립니다.

### Step 7 — PDF 변환 안내 (선택)
인수확인서는 클라이언트 서명 문서라 PDF가 깔끔합니다:
```bash
pandoc ./acceptance/*-인수확인서.md -o ./acceptance/인수확인서.pdf --pdf-engine=xelatex -V mainfont="Apple SD Gothic Neo"
```

### Step 8 — 다음 단계
- 인수확인서 서명 확인 후 `/입금확인 잔금` → `/하자보수 30일` (또는 60일/90일) 실행.
- 자동 종료 알림 예약 가능.
