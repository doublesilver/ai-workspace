# 자동 수정 모드

검수 보고서의 **모든 이슈를 기본값으로 자동 수정**하고 재검증합니다.
인자로 우선순위 필터를 명시하면 해당 범위만 처리합니다.

수정 대상: $ARGUMENTS (지정 없으면 **전체 — Critical + High + Medium + Low**)

## 절차

### Step 1 — 최신 검수 보고서 로드
```bash
ls -t inspection-report-*.md 2>/dev/null | head -1
```
파일이 없으면 → 사용자에게 안내: "검수 보고서가 없습니다. /전수검사부터 실행하세요."

보고서를 읽어 이슈 목록 추출 (기본은 **모두**, 인자로 좁히면 그에 따라):
- 🔴 Critical 항목 모두
- 🟠 High 항목 모두
- 🟡 Medium 항목 모두
- 🟢 Low 항목 모두 (인자 "critical만"·"critical+high"·"critical+high+medium" 등으로 제외 가능)
- 각 이슈의 파일 위치, 문제 설명, 권장 수정안

### Step 2 — 수정 계획 TaskCreate

추출한 이슈 각각을 TaskCreate로 등록 (우선순위 prefix 포함):
```
Task #1: [Critical] [파일:라인] 이슈 제목
Task #2: [High] ...
Task #3: [Medium] ...
Task #4: [Low] ...
```

처리 순서는 항상 **Critical → High → Medium → Low**. 의존성이 있으면 (한 수정이 다른 수정에 영향) addBlockedBy로 순서 설정.

### Step 3 — 자동 수정 실행

각 task에 대해:
1. TaskUpdate status=in_progress
2. 보고서의 권장 수정안에 따라 코드 수정
3. 단위 테스트 또는 빌드로 즉시 검증
4. 통과 시 자동 commit (메시지: `fix: [이슈 제목]`)
5. TaskUpdate status=completed

수정 중 막히면:
- 사용자에게 명확히 보고 (무엇이 막혔는지, 왜)
- 해당 task만 in_progress 유지, 다른 task 계속 진행
- 종료 시 막힌 task 모음 보고

### Step 4 — 재검증 (★ 핵심)

모든 수정 완료 후, **검수를 다시 실행**:
- 처음 검수가 `/전수검사`였으면 → `/전수검사` 다시
- `/검수`였으면 → `/검수` 다시

재검증 결과:
- 새 보고서 생성
- 이전 보고서와 비교:
  - 수정된 이슈: ✅
  - 남은 이슈: ⚠️ (수동 처리 필요)
  - 새로 발견된 이슈: 🆕 (수정 중 회귀 발생)

### Step 5 — 종합 보고

```markdown
## 자동 수정 결과

### 처리한 이슈
- Critical: N건 중 X건 자동 수정 완료
- High: N건 중 Y건 자동 수정 완료
- Medium: N건 중 Z건 자동 수정 완료
- Low: N건 중 W건 자동 수정 완료

### 자동 수정 실패 (사용자 결정 필요)
1. [이슈] — 사유: ...

### 회귀 (수정으로 새로 생긴 이슈)
1. [이슈] — 위치: ...

### 다음 액션
- 남은 Critical/High 0건 → §6 UAT 또는 배포 가능
- 남은 Medium/Low 0건 + 위 조건 → 깨끗한 상태 (출시 준비 완료)
- 또는 남은 이슈 처리 가이드
```

### Step 6 — 자동 commit

자동 수정으로 변경된 모든 파일을 한 번에 commit:
- 메시지: `fix: 자동 수정 — Critical N / High M / Medium K / Low L 해결`
- 본문에 처리한 이슈 목록 (보고서 링크 포함)
- 변경 규모가 크면 우선순위 단위로 commit 분할도 OK (`fix(critical): ...`, `fix(high): ...`, `fix(medium): ...`)

자동 push도 진행 (글로벌 §13.1 정책).

### Step 7 — UAT 결과 stale 확인 (★ v2.0)

자동 수정으로 코드가 변경됐으면 **기존 UAT 결과 파일은 무효화**됩니다 (게이트 ② false sense of security 위험).

- `./uat/*결과*.md` 파일이 있는데 코드 수정이 발생했다면:
  - 사용자에게 알림: "코드가 변경됐습니다. 이전 UAT 결과는 stale 상태입니다. 다음 중 하나 진행 권장:
    1. `/UAT`로 새 시나리오 생성 → 클라이언트 재검수
    2. 영향 범위가 좁다면 기존 결과 파일에서 영향 항목만 ❌ 미통과로 변경 후 재검수"
- 자동수정 보고서에 "UAT 재실행 필요" 명시 항목 추가.

## 무한 루프 방지

자동 수정 → 재검증 → 또 이슈 발견 → 자동 수정 → ... 이 무한히 돌면 안 됨.
**최대 3사이클까지만** 시도. 그 이상은 사용자 개입 필요.
