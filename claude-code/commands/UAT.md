# 6단계 — UAT (User Acceptance Test)

당신은 클라이언트가 직접 검수할 **인수시험 시나리오**를 생성합니다.
**§11 게이트 ②**: UAT 통과 전 Production 배포 금지.

## 절차

### Step 1 — 컨텍스트 수집
- 프로젝트 루트 `CLAUDE.md` 읽기 (기획문서)
- `./reports/` 최신 검수 보고서 읽기 (Critical/High 해결 확인)
- 검수 미해결 항목이 있으면 UAT 진입 금지 → 사용자에게 보고 후 종료

### Step 2 — 인수 시나리오 자동 생성
기획문서 In-Scope 각 항목당 다음 4종 시나리오:

1. **정상 경로** — 핵심 사용자 흐름 (Happy Path)
2. **경계 조건** — 빈 값, 최대 값, 권한 없음
3. **에러 복구** — 네트워크 끊김, 잘못된 입력
4. **모바일/반응형** — 휴대폰에서 동일 흐름

각 시나리오마다:
- 사전 조건 (Given)
- 행동 (When)
- 기대 결과 (Then)
- 통과 기준 (yes/no)

### Step 3 — 클라이언트용 체크리스트 생성

`./uat/YYYY-MM-DD-UAT.md`로 저장. 마크다운 + 모바일에서 보기 쉽게:

```markdown
# {프로젝트명} 인수시험 (UAT) 체크리스트

> 이 체크리스트를 한 항목씩 확인하시고, 문제가 있는 부분은
> "❌ 미통과"로 표시한 뒤 발견 내용을 적어주세요.
> 모든 항목 ✅ 통과 시 Production 배포로 진행합니다.

작성일: YYYY-MM-DD
대상 URL: {preview-url}

---

## 시나리오 1. {In-Scope 항목 1}

### 1-1 정상 경로
- [ ] {Given/When/Then}
- 통과 기준: {...}
- 결과: ⬜ ✅ 통과 / ⬜ ❌ 미통과
- 미통과 시 내용: ___________

### 1-2 경계 조건
...

### 1-3 에러 복구
...

### 1-4 모바일
...

## 시나리오 2. {In-Scope 항목 2}
...

---

## 종합 의견
- [ ] 모든 항목 통과 → Production 배포 진행 OK
- [ ] 일부 미통과 — 위 내용 수정 후 재검수 요청
- 추가 의견: ___________

서명: ___________  날짜: ___________
```

### Step 4 — (선택) 디지털 체크 링크
사용자가 원하면 Google Forms·Tally·Typeform 등 디지털 폼으로 변환 안내.
또는 GitHub Issue 템플릿으로 변환:
```bash
gh issue create --title "UAT — {프로젝트명}" --body-file ./uat/YYYY-MM-DD-UAT.md
```

### Step 5 — 클라이언트 전달용 메시지 생성
한국어 존댓말로 카카오톡·이메일 전달용 메시지:

> 안녕하세요, {프로젝트명} 개발이 마무리되어 인수시험을 요청드립니다.
> Preview URL: {url}
> 체크리스트: {file or link}
> 모든 항목을 한 번 확인하신 뒤 ✅/❌로 답변 부탁드립니다.
> 미통과 항목이 있으면 그 부분만 수정 후 다시 안내드리겠습니다.

### Step 6 — 다음 단계 안내
- "클라이언트 답변을 받으면 §7 결과 파일을 생성한 뒤:"
  - 미통과 0건 → `/배포-production` (hook이 결과 파일 자동 점검)
  - 미통과 1건 이상 → 해당 부분 수정 → `/검수` 재실행 → 다시 `/UAT`

### Step 7 — 결과 파일 표준 (★ §11 게이트 ② 게이트키퍼)

클라이언트 응답이 들어오면 **반드시** 다음 표준 포맷으로 저장합니다:

**파일명**: `./uat/YYYY-MM-DD-결과.md` (정확히 "결과"라는 단어 포함)

**필수 표기 규칙**:
- 통과 항목: `✅ 통과`
- 미통과 항목: **`❌ 미통과`** (정확히 이 문자열, hook이 grep으로 점검)
- 미통과 사유는 항목 바로 아래에 한 줄 이상

```markdown
# {프로젝트명} UAT 결과
작성일: YYYY-MM-DD
수신 채널: 카카오톡 / 이메일 / 미팅
클라이언트 확인: {이름}

## 시나리오 1-1 정상 경로
✅ 통과

## 시나리오 1-2 경계 조건
❌ 미통과
사유: 빈 값 입력 시 에러 메시지가 영어로 나옴

## 시나리오 2-1 ...
✅ 통과

...

## 종합 — 미통과: N건 (★ hook이 이 라인의 N을 추출합니다 — 형식 엄수)
- 총 항목: M개
- 통과: X개
- 클라이언트 추가 의견: ...
```

**중요 — 표준 summary 라인 형식 엄수**:
- hook은 `미통과:\s*[0-9]+` 정규식의 **마지막 매칭**에서 숫자 N을 추출합니다.
- 시나리오 본문에 "❌ 미통과 사유"라는 안내문이 있어도 카운트에 영향 없음 (false positive 방지).
- 결과 파일 끝부분에 `미통과: 0건` 또는 `미통과: 3건` 형식의 라인이 정확히 1개 있어야 합니다.
- 라인이 없으면 hook은 0건으로 간주 → 위험. 반드시 표기.

**hook 차단 조건 (settings.json PreToolUse)**:
- `./uat/*결과*.md` 파일 자체가 없음 → 차단
- summary 라인의 N > 0 → 차단

**해제 방법**:
- 미통과 항목을 모두 해결한 뒤 결과 파일을 **재작성** (기존 결과 파일 갱신 또는 새 파일 추가).
- 가장 최근 파일(`ls -t`)이 기준이라 최신 결과 파일을 만들면 됨.
