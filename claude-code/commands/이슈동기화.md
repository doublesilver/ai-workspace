# GitHub Issues 클라이언트 가시성 동기화

현 외주 프로젝트의 In-Scope 항목을 GitHub Issues로 발행/갱신해, **클라이언트가 Issues 목록으로 진행상황을 실시간 열람**하게 한다 (CCPM 패턴). 내부 추적(§13.4 TaskCreate)을 보완하는 **클라이언트용 외부 창**.

## 선행 점검 (하나라도 실패하면 안내 후 종료)
1. `gh auth status` — GitHub 인증 확인
2. `git remote -v` — GitHub repo 연결 확인 (없으면 "GitHub repo 연결 필요" 안내)
3. 프로젝트 루트 `CLAUDE.md` 존재 — 없으면 "2단계 /기획 먼저" 안내

## 절차
1. **라벨 세트 준비** (멱등):
   ```bash
   bash "$HOME/ai-workspace/claude-code/scripts/gh-labels.sh"
   ```
2. **In-Scope 읽기**: 프로젝트 `CLAUDE.md`의 In-Scope 섹션에서 작업 항목 목록을 추출.
3. **멱등 동기화** (중복 생성 금지):
   - 기존 이슈 조회: `gh issue list --label In-Scope --state all --json number,title --limit 200`
   - In-Scope 항목별로 **같은 제목 이슈가 없을 때만** 생성:
     `gh issue create --title "<항목>" --label In-Scope --body "<수용 기준 / 완료의 정의>"`
   - 이미 있으면 건너뜀.
4. **수용 기준 본문**: §16.4 critical flow에 해당하면, body에 spec-first 수용 테스트 기준(통과 조건)을 적는다. 클라이언트가 "무엇이 완료인지" 명확히 보게.
5. **클라이언트 공유 링크 제시**: `gh repo view --json url` + `/issues?q=is%3Aissue+label%3AIn-Scope` 형태로, 클라이언트가 북마크할 진행상황 URL을 사용자에게 알려준다.

## 진행상황 반영 규칙 (이후)
- 개발 시작 → 해당 이슈 `진행중` 라벨
- 검수/UAT → `검수중`
- 완료(인수) → 이슈 close (또는 커밋/PR 메시지에 `Closes #N` 자동 연동)
- 클라이언트 추가 요구가 범위 밖이면 → `Out-of-Scope` 라벨 + `/CR`로 별도 견적

## 주의 — 클라이언트 적합성 판단
- **GitHub 사용에 익숙한 클라이언트**에 적합. 비기술 클라이언트면 `/클라이언트보고`(마크다운/PDF 보고서)가 더 적절 — 상황 따라 선택.
- **private repo**면 클라이언트를 read 협업자로 초대해야 Issues가 보인다. public이면 링크만 공유.
- ⚠️ **public repo면 In-Scope(클라이언트 요구사항)가 전 세계에 공개**된다. 클라이언트 기밀이면 반드시 private + 협업자 초대로만 운용.
