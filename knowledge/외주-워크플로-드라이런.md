# 외주 워크플로 드라이런 — 시작부터 끝까지 (가상 시나리오)

새 외주를 시작할 때 참조하는 end-to-end 예시. 9단계 + 3게이트 + 슬래시 커맨드 +
모델 전환 + 돌발 상황(계약 회피·스코프 침범·UAT 미통과) + 자기 회고 루프까지 한 바퀴.
모든 등장 클라이언트·프로젝트는 가상이다.

---

## 가상 프로젝트

- **클라이언트:** 동네 베이커리 3지점 운영하는 김사장 (비개발자, 첫 거래)
- **요구:** ① 온라인 픽업 예약 ② 당일 빵 재고 실시간 표시 ③ 단골 적립
- **스택:** Next.js + Supabase + Vercel
- **견적:** ₩680만(VAT 별도), 4주

---

## 0단계 · Discovery (Sonnet)
통화 녹취 → `/인테이크 ./통화.txt` → 요구사항 정리 + "다시 물어볼 5가지".
핵심 질문(결제 방식)이 In/Out-Scope를 가른다. 답변: 현장 결제만, 재고 수동입력,
적립 횟수제, 1지점 먼저, 디자인 맡김.

## 1단계 · 견적·계약 (Opus) — 게이트 ①
`/견적` → In/Out-Scope·견적서·계약서·입금트래커·SLA·책임 5조항 생성.
첫 거래 → 마진 양보 + 다음 의뢰 우선 협상권(가격 전략).

- In-Scope: 픽업 예약, 재고 표시(수동), 횟수제 적립, 1지점, 디자인
- Out-of-Scope: 온라인 결제, POS 연동, 2·3지점, 네이티브 앱
- SLA: 가동률 99.5%, 장애 영업일 4h 응답

**돌발 1 — 계약 회피:** "급하니 계약은 나중에, 바로 시작."
→ 게이트 ①이 차단: 계약/착수금 전 2단계 진입 금지(무상노동·책임 위험).
→ 착수금 입금 → `/입금확인 착수금` → 게이트 해제.

## 2단계 · 기획 (Opus) — Living CLAUDE.md + §16 표면 최소화
`/기획` → 프로젝트 루트 CLAUDE.md, ExitPlanMode 승인.
**§16 적용:** 요구 3개를 화면 3개로 만들지 않는다.
- 화면 통합: "매장 홈"(재고+예약+적립) + "내 예약" = 화면 2개
- 상태 통합: 빈/로딩/에러/품절 패턴 동일화
- critical flow "픽업 예약 생성" = spec-first 수용 테스트 대상
→ 검수 비용·개발 비용 동시 절감.

## 3단계 · 디자인 (Sonnet, 수동)
Claude Design 시안 → `./docs/design/` 저장 → 컴포넌트 트리·토큰 분해. 화면 2개라 부담 최소.

## 4단계 · 개발 (Sonnet, 인증은 Opus)
- `/이슈동기화`: In-Scope를 GitHub Issues로 (private repo + 클라이언트 read 초대) → 진행 실시간 열람
- 인증=Opus, CRUD·UI=Sonnet. 모듈마다 테스트 코드 동반(§15.1), 예약 생성은 spec-first
- GitHub Actions CI(lint→typecheck→test→build) 셋업
- 멀티 클라이언트는 ccmanager worktree로 세션 격리, SessionStart 대시보드 자동 표시

**돌발 2 — 스코프 침범:** "카드결제도 넣어주세요"(Out-of-Scope).
→ `/CR 카드결제 추가` → Opus가 In-Scope 침범 판정 + 별도 견적(첫 거래 후속 할인) →
CR 입금 확인 → 미니 1→5 사이클 별도 진행. **본 계약 일정 보호.**

## 5단계 · 검수 (Sonnet, 보안은 Opus)
`/전수검사` → 8 에이전트 병렬 → High 2(동시성 race, 적립 음수) 등 발견.
`/자동수정` → 전부 수정 + 회귀 테스트(red→green) + CI green → 검수 게이트 통과(Critical/High 0).

## 6단계 · UAT (Sonnet) — 게이트 ②
`/UAT` → 시나리오·체크리스트 생성.
**돌발 3 — 미통과:** "품절 빵도 예약돼요" → 미통과 1건.
→ `/배포-production` 시도 시 게이트 ②가 차단(PreToolUse hook이 vercel --prod 차단).
→ 품절 검증 수정 + 회귀 테스트 → 재시험 → 미통과 0건 → 게이트 해제.

## 7단계 · 배포 (Opus 최종 리뷰) — 게이트 ③
`/배포-preview`(자동) → 확인 → `/배포-production`:
UAT 통과 → 검수 이력 → 기획준수 재확인 → **CI green(§15.2)** → 롤백 계획 → ExitPlanMode.
**게이트 ③:** 사용자 명시 확인 후에만 vercel --prod. 배포 후 health check + 자동 롤백 대기.

## 8단계 · 인수·하자보수 (Haiku 모니터링, Sonnet 픽스)
`/인수` → 인수확인서(소스+tests/+기획+디자인+배포URL, SLA 99.5%, 30일 하자보수, 잔금 청구).
`/입금확인 잔금` → `/하자보수 30일` → UptimeRobot+Sentry 모니터링.
- 하자(무상): "확인 문자 안 옴" = In-Scope 버그 → 무상 수정
- 추가(유상): "2지점 열어주세요" = Out-of-Scope → `/CR`
종료 후 다음 정식 의뢰 우선 협상권 행사.

## 종료 · 자기 회고 루프 (§0 원칙 8)
프로젝트 종료 시 Claude가 비즈니스 언어로 회고:
- 잘 됨: §16 화면 통합으로 시간 -15%
- 마찰: /상태가 src/ 조기 생성 탓에 "4 개발 완료" 오판 → 개선안: S4 판정을 'src 존재'→'커밋 N개 이상'으로
- 게이트 효과: 게이트①이 무상노동 1건, 게이트②가 품절버그 유출 1건 방어
- 판단 필요(디렉터): 휴리스틱 수정? (유지/변경) → "변경" → Claude가 status.sh 수정

---

## 이 드라이런의 교훈
- 게이트 3개가 실제로 막는 지점이 구체적으로 드러난다(계약 전 작업·미통과 배포).
- 돌발 3종(계약 회피·스코프 침범·UAT 미통과)에 대한 표준 대응이 정해져 있다.
- status.sh 휴리스틱 오판 같은 약점도 회고 루프가 잡아 고친다 — "완벽"이 아니라 "개선되는" 시스템.
